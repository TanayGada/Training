## What are SOLID Principles?
1. S -> Single Responsibility Principle
2. O -> Open Closed Principle
3. L -> Liskov Substitution Principle (LSP)
4. I -> Interface Segregation Principle (ISP)
5. D -> Dependency Inversion Principle (DIP) 




### 1. Single Responsibility Principle (SRP) – “One class, one job!”
- A class should have only one reason to change — it should do only one thing.
- Keeping responsibilities separate makes code easier to test, reuse, and maintain.
- Example:
Instead of a ReportManager that handles report creation and saving to disk, split it into ReportGenerator and FileSaver.



### 2. Open/Closed Principle (OCP) – “Open for extension, closed for modification.”
- You should be able to add new functionality without changing existing code.
- Achieved using interfaces, inheritance, or strategy patterns.
- Example:
A Shape interface has a draw() method. You can create new shapes like Circle or Triangle without changing existing Shape-related code.



### 3. Liskov Substitution Principle (LSP) – “Subtypes must work as the base type.”
- Subclasses should behave like their base class — no unexpected behavior or broken logic.
- Promotes reliable inheritance and better polymorphism.
- Example:
If Bird has a method fly(), then Penguin shouldn’t inherit from Bird unless it can also fly. Otherwise, it violates LSP.



### 4. Interface Segregation Principle (ISP) – “Don’t force what you don’t use.”
- Clients shouldn’t be forced to depend on methods they don’t need.
- Prefer multiple small interfaces over a large, all-in-one interface.
- Example:
Instead of one MultifunctionPrinter interface with print(), scan(), fax(), split into Printer, Scanner, and FaxMachine. A basic printer then implements only Printer.



### 5. Dependency Inversion Principle (DIP) – “Depend on abstractions, not concretions.”
- High-level modules should depend on interfaces, not on specific implementations.
- This reduces tight coupling and makes code more testable and extensible.
- Example:
A NotificationSender depends on a MessageService interface. Whether it’s EmailService or SMSService underneath doesn’t matter — it just calls send().


## Additional Principles

### DRY – “Don’t Repeat Yourself”
- Avoid duplicating logic — use functions, classes, or modules to centralize behavior.

### KISS – “Keep It Simple, Stupid”
- Write code that’s simple and clear over clever and complex.
- Easier to read, debug, and maintain.

### YAGNI – “You Ain’t Gonna Need It”
- Don’t build features you don’t currently need.
- Saves time, reduces complexity, and avoids wasted effort.
