# Interface Segregation Principle

## Definition
- No client should be forced to depend on methods it does not use
- Big interfaces should be split into smaller, more specific ones so that classes only need to implement the methods they actually use.

### Example 1:
#### ISP is violated
```java
public interface Worker {
    void work();
    void eat();
}
```
```java
public class HumanWorker implements Worker {
    public void work() {
        System.out.println("Working on tasks...");
    }

    public void eat() {
        System.out.println("Taking a lunch break...");
    }
}

public class RobotWorker implements Worker {
    public void work() {
        System.out.println("Performing automated tasks...");
    }

    public void eat() {
        // Robot doesn't eat!
        throw new UnsupportedOperationException("Robot doesn't eat!");
    }
}
```
What is wrong here?
- RobotWorker is forced to implement eat() even though it has no use for it.
- This violates ISP, because the interface Worker has more methods than necessary for some clients.

##### ISP is followed(Refactored Design)
1. Split the interface into smaller, more relevant ones:
```java
public interface Workable {
    void work();
}

public interface Eatable {
    void eat();
}
```

2. Now classes implement only what they need:
```java
public class HumanWorker implements Workable, Eatable {
    public void work() {
        System.out.println("Human working...");
    }

    public void eat() {
        System.out.println("Human eating...");
    }
}

public class RobotWorker implements Workable {
    public void work() {
        System.out.println("Robot working...");
    }
}
```

### Benefits
1. Classes are not bloated with unnecessary methods.
2. Code is easier to maintain and change.
3. You can evolve interfaces without impacting all classes.


### Example 2: 
#### Bad Design
```java
interface Appliance {
    void turnOn();
    void cook();
    void wash();
}

public class Microwave implements Appliance {
    public void turnOn() { }
    public void cook() { }
    public void wash() {
        throw new UnsupportedOperationException("Microwave can't wash");
    }
}

public class WashingMachine implements Appliance {
    public void turnOn() { }
    public void cook() { 
        throw new UnsupportedOperationException("Washing Machine can't cook");
    }
    public void wash() { }
}
```
What is wrong here?
1. This forces Microwave to have a wash() method
2. This forces WashingMachine to have a cook() method

#### Better Design
```java
interface Switchable {
    void turnOn();
}

interface Cookable {
    void cook();
}

interface Washable {
    void wash();
}

public class Microwave implements Switchable, Cookable {
    public void turnOn() { }
    public void cook() { }
}

public class WashingMachine implements Switchable, Washable {
    public void turnOn() { }
    public void wash() { }
}



```


#### The Interface Segregation Principle helps create a clean, decoupled, modular design. Smaller interfaces reduce dependency noise and makes the codebase more adaptable to future changes.
